\chapter{Introducción y Contexto}

\section{Introducción}

La presente documentación describe el análisis del reto planteado para el desarrollo de una plataforma web de autogestión B2B para Coca-Cola Ecuador. Esta solución busca digitalizar el proceso de pedidos entre la embotelladora y sus comercios aliados, eliminando intermediarios y optimizando la operación comercial mediante tecnología.

La plataforma permite a los comercios realizar pedidos de forma directa, acceder a incentivos automatizados y dar seguimiento a sus entregas en tiempo real, mientras que la embotelladora obtiene trazabilidad completa y herramientas de análisis para la toma de decisiones estratégicas.

\section{Problemática}

El modelo de pedidos actual de Coca-Cola Ecuador presenta las siguientes limitaciones:

\begin{itemize}
  \item Dependencia de vendedores de campo para la captura manual de pedidos, generando demoras y errores.
  \item Costos operativos elevados asociados a comisiones de agentes externos por cada transacción.
  \item Falta de visibilidad en tiempo real del ciclo de pedido, desde la captura hasta la entrega.
  \item Ausencia de un sistema de incentivos estandarizado y automatizado para comercios.
  \item Limitada trazabilidad de datos para la toma de decisiones estratégicas de la embotelladora.
\end{itemize}

\section{Metodología o Estrategia de Desarrollo}

El proyecto adopta una metodología ágil basada en \textbf{Scrum}, con sprints de dos semanas, revisiones continuas y entregas incrementales. La estrategia contempla:

\begin{itemize}
  \item \textbf{Arquitectura mobile-first con soporte offline} mediante sincronización diferida (SQLite local).
  \item \textbf{Integración con ERP/WMS existente} a través de una API REST, evitando duplicación de infraestructura.
  \item \textbf{Desarrollo iterativo} organizado en 6 módulos funcionales independientes pero interconectados.
  \item \textbf{Validación continua} con stakeholders de Coca-Cola Ecuador en cada ciclo de revisión.
\end{itemize}

\section{Análisis de la Solución}

\subsection{Propuesta de Valor Técnica}

La solución propuesta es una plataforma web B2B que centraliza la operación de pedidos. Los pilares técnicos son:

\begin{itemize}
  \item \textbf{Eliminación de la comisión de agentes:} al digitalizar los pedidos se elimina el costo variable por transacción del agente externo, convirtiéndolo en descuentos directos al comercio.
  \item \textbf{Arquitectura mobile-first con soporte offline:} la app opera con sincronización diferida; el comerciante registra pedidos sin conexión y la plataforma los procesa al recuperar la señal.
  \item \textbf{Motor de incentivos automatizado:} calcula descuentos y bonificaciones en tiempo real sin intervención manual.
  \item \textbf{Integración con ERP/WMS existente:} mediante API REST con los sistemas logísticos actuales, sin duplicación de infraestructura.
  \item \textbf{Panel administrativo:} trazabilidad completa del ciclo de pedido con analítica en tiempo real.
\end{itemize}

\section{Historias de Usuario / Casos de Uso}

\subsection{M1 — Registro y Acceso}

El comercio inicia el proceso registrando su cuenta con datos básicos: RUC, nombre comercial, dirección y contacto. El sistema envía obligatoriamente un código OTP por SMS para verificar la identidad antes del primer acceso. Para accesos posteriores el sistema incluye autenticación de credenciales. En caso de olvido de contraseña, el flujo de recuperación incluye verificación OTP como paso obligatorio.

\subsection{M2 — Catálogo de Productos}

El comercio accede al catálogo y el sistema carga productos desde el backend mostrando imagen, precio vigente y disponibilidad. Sin conexión a internet, el catálogo se carga desde caché local en modo offline. El administrador puede modificar precio y disponibilidad, lo que dispara sincronización inmediata en la app del comercio.

\subsection{M3 — Pedidos}

El comercio registra un pedido en tres pasos secuenciales obligatorios: (1) seleccionar productos y cantidades, (2) calcular el descuento en tiempo real, (3) confirmar la orden. Sin conexión, el pedido se guarda en SQLite local y se sincroniza automáticamente al recuperar la señal. El comercio puede consultar el historial y realizar reórdenes directas.

\subsection{M4 — Incentivos y Recompensas}

El sistema aplica dos mecanismos de incentivo:

\begin{itemize}
  \item \textbf{Descuentos por volumen:} umbrales configurables desde el panel (ej.\ 5\% desde \$200, 10\% desde \$500). El descuento se refleja en tiempo real antes de confirmar.
  \item \textbf{Bonificación por frecuencia:} los comercios que realizan pedidos dentro de una ventana de tiempo predefinida acumulan puntos canjeables, evaluados automáticamente por el sistema.
\end{itemize}

Adicionalmente el sistema implementa un \textbf{esquema de Categoría Comercial}, basado en el volumen acumulado de los últimos 90 días:

\subsection{M5 — Logística y Despacho}

Una vez confirmado un pedido, el sistema asigna automáticamente la ruta evaluando la ubicación geográfica del comercio y la capacidad del vehículo. El comercio recibe notificaciones por app y SMS en cada cambio de estado. La entrega se registra mediante firma digital o código de verificación.

\subsection{M6 — Administración y Reportes}

El administrador accede al panel con autenticación MFA. Desde el panel puede visualizar datos en tiempo real, configurar reglas de incentivos sin asistencia técnica y generar reportes exportables en CSV o PDF filtrados por período, comercio o zona geográfica.

\subsection{Preguntas y Respuestas sobre el Diagrama}

\begin{enumerate}
    \item \textbf{¿Qué pasa desde que un comercio hace un pedido hasta que se le asigna una ruta de entrega?} \\
    Cuando un comercio presiona "Confirmar pedido", el sistema primero accede a la tabla COMERCIO para verificar que el comercio existe, está activo y obtener su dirección de entrega. Luego crea un nuevo registro en PEDIDO vinculado al comercio\_id y al usuario\_id del usuario que está logueado en ese momento. Inmediatamente después, por cada producto seleccionado se crean registros en DETALLE\_PEDIDO, cada uno con su pedido\_id, producto\_id, cantidad y precio unitario tomado en ese instante de PRODUCTO. Con todos los detalles registrados, el motor de incentivos consulta INCENTIVO filtrando por comercio\_id y el monto total del pedido para aplicar el descuento correspondiente, que se escribe de vuelta en el campo descuento de PEDIDO. Finalmente el servicio de logística busca en RUTA la zona que coincide con la dirección del comercio y tiene capacidad disponible, y crea un nuevo registro en ENTREGA enlazando el pedido\_id con el ruta\_id encontrado. En total interactúan 6 tablas: COMERCIO, USUARIO, PEDIDO, DETALLE\_PEDIDO, PRODUCTO, INCENTIVO, RUTA y ENTREGA.

    \item \textbf{¿Cómo sabe el sistema qué descuento aplicarle a un comercio específico en tiempo real antes de confirmar el pedido?} \\
    Cuando el comercio está armando su carrito, el sistema toma el comercio\_id de la sesión activa en USUARIO y con él va a COMERCIO para leer el campo categoria y volumen\_90d. Con esa categoría va a INCENTIVO y filtra todos los registros donde comercio\_id coincide y el monto actual del carrito cae entre umbral\_min y umbral\_max, además de verificar que activo sea verdadero. De los registros que devuelve busca el de mayor descuento\_pct aplicable. Paralelamente recorre cada ítem del carrito consultando PRODUCTO para obtener el precio vigente de cada uno, multiplica por la cantidad y suma el subtotal. Sobre ese monto aplica el porcentaje encontrado en INCENTIVO y muestra el resultado en pantalla antes de que el comercio confirme. Si el comercio cambia una cantidad, el ciclo se repite completo. Interactúan 4 tablas: USUARIO, COMERCIO, INCENTIVO y PRODUCTO.

    \item \textbf{¿Cómo se construye el historial completo de un comercio mostrando cada pedido con sus productos, estado de entrega y notificaciones recibidas?} \\
    El sistema toma el comercio\_id del comercio autenticado y va a PEDIDO para traer todos los registros donde ese comercio\_id coincide, ordenados por fecha descendente. Por cada pedido encontrado hace dos cosas en paralelo. Primero va a DETALLE\_PEDIDO filtrando por pedido\_id para obtener todas las líneas de productos, y por cada línea consulta PRODUCTO usando el producto\_id para obtener el nombre e imagen actual. Segundo, busca en ENTREGA el registro con ese pedido\_id para leer fecha\_estimada, fecha\_real y tipo\_confirmacion, y desde ahí va a RUTA con el ruta\_id para mostrar la zona y el vehículo asignado. Adicionalmente consulta NOTIFICACION filtrando por pedido\_id para listar todos los mensajes que se enviaron en ese ciclo, con su canal y fecha. Todo se agrupa por pedido y se presenta en una sola vista. Interactúan 7 tablas: COMERCIO, PEDIDO, DETALLE\_PEDIDO, PRODUCTO, ENTREGA, RUTA y NOTIFICACION.

    \item \textbf{¿Cómo actualiza el sistema automáticamente la categoría comercial de un comercio al cierre del ciclo de 90 días?} \\
    Al dispararse el cierre de ciclo, el sistema recorre todos los registros de COMERCIO que tienen estado activo. Por cada uno toma su id y va a PEDIDO filtrando por comercio\_id y por fecha dentro de los últimos 90 días, sumando todos los monto\_total de pedidos con estado "entregado". Ese valor acumulado lo compara contra los umbrales definidos en INCENTIVO filtrando los registros de tipo "categoria" y activo verdadero para determinar si el comercio sube, mantiene o baja de categoría entre Estándar, Preferente y Estratégico. Con la categoría nueva calculada vuelve a COMERCIO y actualiza los campos categoria y volumen\_90d con el valor sumado. Si la categoría cambió, el servicio de notificaciones crea un nuevo registro en NOTIFICACION con comercio\_id, un mensaje informando el nuevo nivel y el canal correspondiente. Interactúan 3 tablas: COMERCIO, PEDIDO e INCENTIVO, con escritura de vuelta en COMERCIO y creación en NOTIFICACION.

    \item \textbf{¿Cómo genera el panel administrativo un reporte de pedidos filtrado por zona geográfica, período y estado de entrega, con los datos completos de cada comercio involucrado?} \\
    El administrador selecciona una zona, un rango de fechas y un estado de entrega en el panel. El sistema primero va a RUTA y obtiene todos los id de rutas cuya zona coincide con el filtro seleccionado. Con esos ruta\_id va a ENTREGA y filtra todos los registros que los contengan y cuya fecha\_estimada esté dentro del rango de fechas ingresado. De cada entrega toma el pedido\_id y va a PEDIDO para verificar que el estado coincida con el filtro y para obtener monto\_total, descuento y fecha. Desde cada pedido sube a COMERCIO usando comercio\_id para obtener nombre comercial, RUC, dirección y categoría actual. Luego baja a DETALLE\_PEDIDO filtrando por pedido\_id para contar cuántos productos distintos tuvo ese pedido, y consulta PRODUCTO por cada producto\_id para mostrar los nombres. Finalmente consulta NOTIFICACION por pedido\_id para incluir en el reporte el log de comunicaciones del ciclo. Todo se agrupa, se suma por comercio y por zona, y se exporta. Interactúan 7 tablas: RUTA, ENTREGA, PEDIDO, COMERCIO, DETALLE\_PEDIDO, PRODUCTO y NOTIFICACION.
\end{enumerate}
